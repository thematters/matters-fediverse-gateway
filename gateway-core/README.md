# Matters Gateway Core

`gateway-core` is the dynamic ActivityPub runtime for Matters Fediverse Gateway. It complements the static IPFS/IPNS-oriented publishing path by handling the stateful parts of federation: discovery, signed inbox traffic, followers, outbound delivery, moderation controls, persistence, and recovery.

## Runtime Scope

- Inbound `Follow`
- Inbound `Create`, `Reply`, `Like`, `Announce`, and `Undo`
- Outbound `Create`, `Like`, `Announce`, `Update`, and `Delete`
- Outbound `Accept` and `Reject`
- WebFinger, host-meta, NodeInfo, actor, outbox, followers, and following endpoints
- HTTP Signature verification and signing
- Remote actor discovery, cache, and key refresh
- Static outbox bridge for output generated by `ipns-site-generator`
- Domain block, actor suspension, remote actor policy, abuse queue, audit log, and evidence retention baselines
- Legal takedown, manual replay, retry, and dead-letter controls
- Instance-level and actor-level rate limits
- SQLite persistence, backup, restore, reconcile, and runtime health checks
- Structured metrics, alerts, logs, webhook dispatch, and Slack incoming webhook support
- Thread reconstruction, mention mapping, local conversation projection, and remote `acct:` mention resolution

Paid, private, encrypted, and message-like content must stay outside the federation boundary.

## Quick Start

```bash
cd gateway-core
npm install
npm start
```

The default runtime reads `./config/dev.instance.json` and writes state to `./runtime/dev-state.sqlite`.

`config/dev-private-key.pem` and `config/dev-public-key.pem` are development-only keys. Do not use them for a real instance.

## Boundary With ipns-site-generator

`ipns-site-generator` remains the static publisher. It should generate public article pages, public feed files, and a static ActivityPub seed bundle for content that is already safe to federate.

`gateway-core` remains the dynamic gateway. It should read that static seed, rewrite it to the canonical instance actor, expose live discovery endpoints, accept signed inbox traffic, maintain followers state, deliver outbound activities, and provide moderation and recovery tooling.

The bridge is configured per actor:

```json
{
  "actors": {
    "alice": {
      "displayName": "Alice Matters",
      "staticOutboxFile": "../fixtures/alice-outbox.jsonld",
      "publicKeyPemFile": "./dev-public-key.pem",
      "privateKeyPemFile": "./dev-private-key.pem"
    }
  }
}
```

See [`../docs/ipns-gateway-cloudflare-plan.md`](../docs/ipns-gateway-cloudflare-plan.md) for the proposed integration contract and deployment path.

## Config Shape

`config/dev.instance.json` shows the minimum configuration shape.

- `instance`: canonical domain, brand metadata, NodeInfo metadata
- `actors`: local actors, profile data, key material, follow policy, and optional `staticOutboxFile`
- `remoteActors`: optional seed data for remote actors
- `remoteDiscovery`: cache TTL and live discovery behavior
- `delivery`: retry limits, outbound user agent, and processing lease timeout
- `moderation`: domain blocks, actor suspensions, remote actor policies, evidence retention, and rate limits
- `runtime`: store driver, file paths, SQLite path, alerting, metrics dispatch, logs dispatch, and external sinks

## SQLite Backup

```bash
cd gateway-core
npm run backup:sqlite
```

This creates a consistent backup of the configured SQLite runtime and writes a manifest next to the backup file.

## SQLite Restore

```bash
cd gateway-core
npm run restore:sqlite -- --input-file ./runtime/backups/dev-state-<timestamp>.sqlite --target-file ./runtime/dev-state.sqlite
```

This restores a selected backup into a target SQLite runtime. By default it creates a pre-restore backup of the existing target first.

## Runtime Alerts

```bash
cd gateway-core
npm run dispatch:alerts -- --output-file ./runtime/alerts/latest.json
```

This reads runtime metadata, metrics, and alerts, then writes a JSON bundle for an external scheduler or monitoring system.

To send the bundle to a webhook sink:

```bash
cd gateway-core
npm run dispatch:alerts -- \
  --output-file ./runtime/alerts/latest.json \
  --webhook-url http://127.0.0.1:8788/runtime-alerts \
  --webhook-header x-alert-source=gateway-core \
  --webhook-bearer-token <token>
```

Slack incoming webhooks can use the same alert dispatch flow:

```bash
cd gateway-core
npm run dispatch:alerts -- \
  --output-file ./runtime/alerts/latest.json \
  --slack-webhook-url https://hooks.slack.com/services/xxx/yyy/zzz \
  --slack-channel '#gateway-alerts' \
  --slack-username matters-gateway \
  --slack-icon-emoji :satellite:
```

`POST /admin/runtime/alerts/dispatch` supports the same webhook and Slack fields. It writes sink type, target host, and response status to audit and trace records.

## Observability Drill

```bash
cd gateway-core
npm run drill:observability
```

This produces alerts, metrics, and logs bundles under `./runtime/drills/`, then attempts to send each bundle to its configured external sink.

Manual staging drill:

```bash
cd gateway-core
node scripts/run-staging-observability-drill.mjs \
  --config ./config/dev.instance.json \
  --output-dir ./runtime/drills/manual-check \
  --require-sinks
```

The output directory contains `alerts.json`, `metrics.json`, `logs.json`, and `report.json`.

## Local Sandbox Interop

```bash
cd gateway-core
npm run check:local-sandbox
```

This starts a local sandbox actor, checks WebFinger, actor, and outbox discovery, then sends a signed `Follow` to verify that the gateway can return an `Accept`.

## Mastodon Sandbox Interop

```bash
cd gateway-core
MASTODON_BASE_URL="https://mastodon.example" \
MASTODON_ACCESS_TOKEN="<token>" \
GATEWAY_PUBLIC_BASE_URL="https://gateway.example" \
npm run check:mastodon-sandbox
```

This reuses the local sandbox acceptance fields against a real Mastodon instance: remote account resolution, follow, and relationship polling.
